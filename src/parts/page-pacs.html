Inbound API contract

        

          The production listener exposes a single ingest endpoint. Each PACS connection points its outbound feed at the URL below with an Authorization header derived from the connection's API key.
        

        
POST /api/pacs/ingest
Authorization: Bearer <connection_api_key>
Content-Type: application/json   (or text/hl7 for raw HL7)

Body — one of:
{
  "format": "hl7"    | "fhir" | "plain",
  "payload":  <string for hl7/plain, object for fhir>,
  "source":   "fluency-riverview",
  "received_at": "2026-05-13T15:00:00Z"
}

Response 200 OK
{
  "report_id": 4231,
  "accession": "ACC-001234",
  "patient_mrn": "MRN-0009",
  "coding_url": "/api/reports/4231"
}


      


      

        
Connection schema

        

          Each connection persists as a row. Swap the localStorage adapter for Supabase by enabling RLS on this table.
        

        
create table pacs_connections (
  id          uuid primary key default gen_random_uuid(),
  name        text not null,
  vendor      text not null,   -- fluency|powerscribe|epic_fhir|hl7_listener|webhook|folder|s3
  config      jsonb not null,
  api_key     text not null,   -- bearer token the vendor sends with each post
  status      text not null default 'configured', -- configured|active|paused|error
  last_sync   timestamptz,
  last_error  text,
  practice_id uuid references practices(id),
  created_at  timestamptz default now()
);

create table pacs_inbound (
  id            bigserial primary key,
  connection_id uuid references pacs_connections(id),
  format        text not null,
  payload       jsonb,
  raw           text,           -- raw HL7 / FHIR / text for audit
  accession     text,
  patient_mrn   text,            -- de-identified before storage in V1
  report_id     bigint references reports(id),
  received_at   timestamptz default now()
);

      

    


    

      
Vendor-specific wiring notes

      
M*Modal / Solventum Fluency for Imaging
      

        
    
          
  • Fluency emits an HL7 v2 ORU^R01 message on signing — point its outbound MLLP feed at /api/pacs/ingest wrapped in an HL7-over-HTTPS adapter, OR run our MLLP receiver on the Fluency network.
    • 
          
  • For Fluency Catalyst (cloud): poll the FHIR DiagnosticReport endpoint instead. Request read scope.
    • 
          
  • OBX|TX segments carry the report body (one line per OBX). Our parser concatenates these in order.
    • 
          
  • MSH-10 → accession; PID-3 → patient_mrn (de-identify before storage in V1).
    • 
        

      

      

      
Nuance Powerscribe 360
      

        
    
          
  • Powerscribe stores signed reports in its SQL backend. Easiest path: ODBC read against the Order + Report tables joined on accession, scheduled every 5 min.
    • 
          
  • Alternative: enable the Powerscribe Outbound API (SOAP) and have it POST to our webhook for each finalized report.
    • 
          
  • For shops on Powerscribe One (cloud): use the FHIR DiagnosticReport endpoint via the Catalyst gateway.
    • 
        

      

      

      
Epic Radiant + Cogito
      

        
    
          
  • Two options: poll FHIR R4 /DiagnosticReport?category=RAD&status=final&_lastUpdated=gt<cursor> or subscribe via SMART backend services (Bulk Data Export).
    • 
          
  • OAuth2 client-credentials grant — Epic's Backend Services documentation covers JWT assertion.
    • 
          
  • Body content lives in presentedForm[].data (base64) — typically RTF or HTML; strip tags before passing to the coder.
    •
